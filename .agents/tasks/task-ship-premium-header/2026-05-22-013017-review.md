# Premium Shopify Header - Semantic Review (v1)

A single squashed commit lands the entire "Premium Header" surface (Phases 1-5)
on top of stock Dawn: a 1352-line `sections/header.liquid`, a 809-line
`assets/header-premium.css`, a 179-line `assets/header-premium.js`, and a
46-setting schema gated behind one master toggle (`hp_enable_premium`,
default `false`). The premium CSS and JS only `<link>`/`<script>` in when
the master toggle is on, and every visual rule is scoped under
`.header-wrapper--premium`, so default Dawn is preserved when the toggle
is off. A self-contained Python validator and a reality-check report
ship alongside the code.

**Watch for:**
- **[confirmed]** Bottom-nav cart count never updates after add-to-cart - the JS listens for `cart:refresh` / `cart-update` DOM events that Dawn does not dispatch (Dawn uses the `PUB_SUB_EVENTS.cartUpdate` pubsub broker).
- **[confirmed]** Mobile drawer "push" animation has no open-state transform; selecting it leaves the drawer in its closed-state translate.
- **[confirmed]** The reality-check report claims coverage was verified against `--hp-foo` and `data-hp-foo` derived forms; `validate.py` only counts the literal `hp_foo` token. Coverage still passes (every id appears at least once in the Liquid body), but the report overstates what the validator proves.
- **[likely]** `body:has(.hp-bottom-nav) { padding-bottom: 64px }` unconditionally pads any merchant-installed sticky footer up by 64px on mobile when the bottom nav is on.
- **[likely]** Premium cart-bubble background defaults to `#000` and overrides Dawn's color-scheme accent the moment the master toggle flips on, regardless of whether the merchant touched `hp_cart_badge_bg`.

## High-level view

The change is structurally additive: zero stock Dawn schema ids, `t:`
keys, or settings types are altered. All 14 original Dawn ids and every
`t:` translation reference round-trip identically between the cached
`dawn-header.liquid` and the new `sections/header.liquid`. With the
master toggle off the premium asset tags don't render, the `--hp-*`
variables aren't emitted, the wrapper modifier class isn't added, and
none of the new `hp-*` markup is rendered.

The premium "API surface" is implemented as a CSS-variable + data-attribute
contract: the `{% style %}` block emits `--hp-*` custom properties on
`.section-header`, and the wrapper picks up a row of `data-hp-trigger /
-anim / -text-transform / -hover-style / -cart-badge / -cart-badge-pos /
-mobile-drawer / -mobile-anim / -mobile-text-transform / -smooth-scroll /
-reduced-motion` attributes. The CSS keys all variants off these
attributes, so a future merchant choice the CSS doesn't recognize silently
no-ops, which is the right failure mode. New labels and `info` strings,
however, are hardcoded English instead of `t:` keys, so non-English
merchants see English in the theme editor for the premium surface only.

The JS is a small IIFE that finds `.header-wrapper--premium`, optionally
attaches hover-open handlers to Dawn's `<details>`-based dropdowns
(gated to `(hover: hover)` and `>=990px`), marks `.section-header` with
`data-hp-has-transparent` so a sibling-selector CSS rule can zero out
the next section's top margin, and wires the bottom-nav search button
to click Dawn's existing `details-modal summary`. Two real defects live
in this file: cart-count sync that listens for events Dawn never
dispatches, and `shopify:section:unload` cleanup that doesn't tear down
the `document`-level cart listeners. The transparent-header offset
likewise depends on the JS having run; if the script fails to load,
the next section overlaps the floating header.

The mobile drawer position/animation rules in CSS are partial. The
"left" drawer-position option and the "slide" animation option have
no rules at all and behave correctly only because Dawn's defaults
happen to match the labels. The "push" animation declares a
`transition` on the drawer's `transform` but never declares an
open-state transform value, so when push is selected the drawer
stays at its initial closed translate and never animates open.
The "fade" rule declares `transform: none` on the open state, which
clobbers Dawn's `translateX` close-path animation - the drawer
fades but doesn't slide back.

The bundled `validate.py` is real, not decoration. It parses
`{% schema %}` as JSON, asserts exactly 46 `hp_*` ids, checks each
range for `(max-min)/step <= 100` and `(default-min) % step == 0`
with a float-safe modulo, balances ten paired Liquid tags
(correctly excluding `{% liquid %}`, which has no `endliquid`),
confirms each `hp_*` id appears at least once in the corpus, and
flags bareword `null` and `width="auto"`. The weak spot is that the
coverage check only matches the literal `hp_foo` token, not the
derived `--hp-foo` or `data-hp-foo` forms. The reality-check report
describes a stronger coverage check than the validator actually
performs; the per-id table in the report shows `css=0, js=0` for
every single id for the same reason.

### Master toggle and gating discipline

The premium feature surface is gated at three layers: asset loading
(`<link>` and `<script>` only emitted inside
`{% if section.settings.hp_enable_premium %}`), CSS-variable definitions
(the `--hp-*` block in `{% style %}` is inside the same condition), and
DOM emission (the `header-wrapper--premium` modifier class, the row of
`data-hp-*` attributes, the top line, the bottom nav, the wishlist
icon, the inline search, and the alt logos all sit inside
`hp_enable_premium` checks). The transparent-header path layers a
second gate (`hp_enable_transparent` plus the page-type case statement
on `hp_transparent_apply_on`) before activating. Backward compatibility
is therefore not just "the styles aren't loaded" - the premium markup
itself doesn't render, so screen readers don't traverse extra DOM
nodes when the toggle is off.

### CSS variable + data-attribute contract

The variant matrix lives entirely in attribute selectors keyed off the
wrapper - five `data-hp-text-transform` values, four
`data-hp-hover-style` values, four `data-hp-anim` values, three
`data-hp-cart-badge`, two `data-hp-cart-badge-pos`, four
`data-hp-mobile-drawer`, three `data-hp-mobile-anim`, three
`data-hp-mobile-text-transform`. Two values land on a no-op CSS path
today: `hp_mobile_drawer_position == "left"` has no matching rule and
inherits Dawn's stock left-slide drawer; `hp_mobile_menu_anim == "slide"`
similarly has no rule. Both behave correctly only because Dawn's
defaults happen to line up with the labels. If Dawn changes its
drawer behavior, these options become silently broken.

The schema labels and `info` strings for all 46 new ids are hardcoded
English ("Enable Premium Header", "Top line height", "Search bar
placeholder", etc.) rather than `t:` keys, so non-English merchants
see English for the new settings only. Stock Dawn strings remain
`t:`-keyed; the regression is contained to the new surface.

### Mobile drawer animations - "push" and "fade" defects

The `data-hp-mobile-anim="push"` block declares
`transition: transform var(--hp-anim-duration) cubic-bezier(...)` on
`.menu-drawer` and a `body:has(...)` rule that locks scroll when the
drawer is open. It does not declare a visible-state transform.
Right/fullscreen variants both pair their closed-state transform with
a `menu-drawer[open] .menu-drawer { transform: ... }` open state.
Push is missing this:

```css
/* Push animation: drawer pushes page content */
.header-wrapper--premium[data-hp-mobile-anim="push"] .menu-drawer {
  transition: transform var(--hp-anim-duration, 300ms) cubic-bezier(0.4, 0, 0.2, 1);
}
body:has(.header-wrapper--premium[data-hp-mobile-anim="push"] menu-drawer[open]),
body:has(.header-wrapper--premium[data-hp-mobile-anim="push"] details[open] > .menu-drawer) {
  overflow: hidden;
}
```

When this is the only drawer animation rule, the drawer's transform is
whatever Dawn leaves it at. With "right" or "fullscreen" position
selected, the closed-state transform is set by the position rule;
"push" then animates to that closed value without ever animating to
open. With the default position, Dawn's stock open-state transform
applies and "push" effectively does nothing visible.

The fade variant has the inverse problem on the close path. Its open
rule is `opacity: 1; transform: none`, where `transform: none`
overrides Dawn's `translateX` and prevents the drawer from sliding
out when it closes - the drawer fades to zero opacity in place.

### Transparent header path depends on JS

The "push body content below absolute header on transparent pages" rule
is implemented as
`.shopify-section.section-header[data-hp-has-transparent="true"] + * { margin-top: 0 !important; }`,
which depends on the JS having run to set `data-hp-has-transparent` on
`.section-header`. If JS fails to load (extension blocking, CSP issue),
the next section's top margin isn't zeroed and the transparent header
overlaps it. A CSS-only fallback (keying off `.header-wrapper--transparent`
as an ancestor selector instead of a sibling-of-section data attribute)
is not implemented. Worth flagging because the rest of premium
gracefully degrades to "Dawn behavior" when JS fails - this rule
specifically degrades to "broken layout".

### Bottom-nav cart count is wired to events that don't fire

```js
syncBottomNavCartCount();
document.addEventListener('cart:refresh', syncBottomNavCartCount);
document.addEventListener('cart-update', syncBottomNavCartCount);
```

Dawn does not dispatch DOM-level `cart:refresh` or `cart-update`
events. Cart updates flow through Dawn's `PubSub` broker
(`PUB_SUB_EVENTS.cartUpdate` and friends in `assets/pubsub.js` and
`assets/global.js`). The initial `syncBottomNavCartCount()` call sets
the count correctly on page load by reading `#cart-icon-bubble`.
Subsequent add-to-cart, cart-drawer interactions, quantity changes,
and notification updates do not trigger either listener, so the
bottom-nav count stays at its load-time value until the page reloads.

The fix is either `subscribe(PUB_SUB_EVENTS.cartUpdate, ...)` (requires
importing the broker) or a `MutationObserver` on `#cart-icon-bubble`
that mirrors any text change into `.hp-bottom-nav__cart-count`. The
observer approach has the advantage of not depending on the pubsub
API surface.

### Lifecycle: section reload and listener cleanup

`shopify:section:load` re-runs `initPremiumHeader()` end-to-end.
`setupBottomNav()` re-subscribes its `document`-level cart listeners on
every load with no removal in `unload`. Today this is a no-op because
the events never fire; the day someone fixes the cart-sync bug, this
becomes a duplicate-handler bug unless cleanup is added. The hover-handler
`WeakMap` keys on parent `<li>` nodes that get GC'd when the old DOM is
removed, so re-init doesn't fight itself there. `data-hp-initialized` is
set on the wrapper but never read - looks like an abandoned idempotency
marker that should be removed or actually consulted.

### Bottom-nav search button targeting

```js
var searchSummary = document.querySelector('.section-header details-modal summary');
if (searchSummary) searchSummary.click();
```

This grabs the first `details-modal summary` inside `.section-header`
in DOM order. Dawn renders search modals in two places: once before
the menu/logo when `logo_position == 'top-center'` or `menu == blank`
(id `Search-In-Modal-1`), and once inside the icons block (id
`Search-In-Modal`). With premium + inline search on, the icons-block
render is wrapped in `medium-hide large-up-hide` so it's mobile-only.
The bottom nav is mobile-only, so the click finds either modal and
opens it. Positionally fragile: if Dawn ever adds another
`details-modal` to the section header (a localization modal, an
account modal), the bottom-nav search button could open the wrong
thing. Pinning to `[id^="Search-In-Modal"]` would be safer.

### Bottom-nav padding side effect

```css
@media screen and (max-width: 749px) {
  body:has(.hp-bottom-nav) {
    padding-bottom: 64px;
  }
  ...
}
```

On mobile, when the bottom nav is on, the body gets 64px of bottom
padding so the last section sits above the fixed bar. Anything else
that pins to `bottom: 0` on mobile - sticky footers, chat widgets,
cookie banners - now sits 64px above the viewport bottom and the
bottom nav floats over it.

### Cart-bubble color override always fires

```css
.header-wrapper--premium .cart-count-bubble {
  background-color: var(--hp-cart-badge-bg, #000);
  color: var(--hp-cart-badge-text, #fff);
  ...
}
```

The default values for `hp_cart_badge_bg` (`#000000`) and
`hp_cart_badge_text` (`#FFFFFF`) are emitted into the `--hp-*`
variables the moment premium turns on, regardless of whether the
merchant actually styled the badge. Stock Dawn paints
`.cart-count-bubble` with the color-scheme accent, so flipping
premium on (with no other changes) flips the cart badge from the
theme accent to flat black-on-white. The master-toggle "default
Dawn behavior unless configured" framing doesn't quite predict this.

### Validator vs reality-check

`validate.py` does seven checks and none of them silently fail-open:
schema parses as JSON; exactly 46 `hp_*` ids; range invariants
(numeric not stringly-typed, `(max-min)/step <= 100`,
`(default-min) % step == 0` with a `1e-9` epsilon both above zero
and above `step` - the latter handles the `5.0 % 5.0 == 5.0`
floating-point edge); ten paired Liquid tags balance (excluding
`{% liquid %}`, which has no `endliquid`); each `hp_*` id appears
at least once in `(liquid_body \ schema_region) + css_text + js_text`;
no bareword `null`; no `width="auto"`. The float-safe modulo correctly
handles `step=0.5`-style values (none ship today, but the math is right).

The weak spot is the coverage check. The reality-check report claims
coverage was verified by counting both the literal `hp_foo` token and
the derived `--hp-foo` and `data-hp-foo` forms. The validator only
counts the literal token. Coverage still passes because every id is
written into the `{% style %}` and wrapper-attribute blocks at least
once - the strict literal check is a sufficient subset of what the
report describes. A fabricated `hp_foo` id with no Liquid wiring would
still fail the literal check, so the gap doesn't admit false
positives. But the per-id table in the report shows `css=0, js=0`
for every single id (because `hp_foo` literal genuinely doesn't appear
in CSS/JS - they reference `--hp-foo` and `data-hp-foo`); a reader of
the table could be excused for thinking nothing is wired through CSS
or JS, when in reality the wiring is just one indirection away.

### Not tested

There is no Shopify rendering harness available, so no runtime-DOM
assertions. Within the static toolchain that exists:

- The validator does not check that `--hp-foo` / `data-hp-foo` derived
  forms appear in CSS for ids that should be CSS-driven (e.g.
  `hp_top_line_color`, `hp_anim_duration`). A schema id that was
  removed from the CSS but kept in the Liquid emission would pass
  validation.
- No check that the JS reads a setting it claims to consume
  (`hp_dropdown_trigger` is read via the `data-hp-trigger` attribute -
  the validator can't distinguish a read from an absent read).
- No DOM-level assertion that with the master toggle off, none of the
  `hp-*` classes or `data-hp-*` attributes appear in rendered output.
  The Liquid gating is correct on inspection, but a regression test
  would catch a future un-gated emission.

### Bundled task-state in the commit

The commit ships `.gitignore` (one line), the orchestrator's
`task.json` / `context.json` / `features/FEAT-001.json`,
`reality-check.md`, `validate.py`, and `validator-output.txt`
under `.agents/tasks/...`. Shopify's theme uploader ignores
non-theme directories, so this won't break the upload, but it does
bloat the commit. A follow-up could add `.agents/` to a
`.shopifyignore` or move the task tree out of the theme entirely.
