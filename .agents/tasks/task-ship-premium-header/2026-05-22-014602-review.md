# Premium Shopify Header - Semantic Review (v2)

A follow-up commit (`2e2b5a0`) takes the 10 issues from the v1 review and
ships fixes for nine of them, with the tenth (English schema labels)
deferred per the user's explicit "no `t:` keys for new custom settings"
rule. The cart-count desync moves from a never-fires DOM event to a
`MutationObserver` on `#cart-icon-bubble`. The mobile drawer "push"
animation gets the missing open-state transform; "fade" loses the
`transform: none` that was clobbering Dawn's close path. A new
`hp_cart_badge_override` checkbox (default `false`) gates the
cart-bubble color override, bumping the hp_ count to 47 and keeping
Dawn's color-scheme accent intact when premium is flipped on with no
other configuration. The transparent-header offset gets a CSS-only
`:has()` fallback so layout still works without JS. The validator
coverage check now counts all three forms (`hp_foo`, `--hp-foo`,
`data-hp-foo`), aligning the per-id table in the report with reality.

**Watch for:**
- **[confirmed]** Cart-count desync is only partially fixed. When the
  cart is empty at server render, the `.hp-bottom-nav__cart-count` span
  isn't emitted (gated on `{%- if cart != empty -%}`), so the observer
  fires on add-to-cart but `syncBottomNavCartCount()` early-returns
  forever - the bottom-nav count stays hidden until full page reload.
- **[confirmed]** New collision: with `hp_mobile_menu_anim="push"` AND
  `hp_mobile_drawer_position="fullscreen"`, the push fix's open-state
  `transform: translateX(0)` overrides fullscreen's `translateY(0)` by
  source order, so the drawer animates between mismatched axes
  (`translateY(-100%)` to `translateX(0)`).
- **[likely]** The new `hp_cart_badge_override` checkbox sits BELOW
  `hp_cart_badge_bg` and `hp_cart_badge_text` in the schema. A merchant
  tweaking the colors first sees no effect and has to scroll past to
  find the gate.

## High-level view

The cart-count plumbing is correct: a module-scoped MutationObserver on
the persistent `#cart-icon-bubble` element with childList + subtree +
characterData, paired with `teardownPremiumHeader()` called on both
`shopify:section:load` (before re-init) and `shopify:section:unload`.
The fix lands one level too low - the receiver `.hp-bottom-nav__cart-count`
span is server-rendered only when `cart != empty`, so the empty-cart
boot followed by the first add-to-cart has no element for the observer
to update.

The mobile drawer push fix adds `transform: translateX(0)` as the
open state. This pairs correctly with default, left, and right drawer
positions but breaks fullscreen, where the closed state is on the
Y axis (`translateY(-100%)`). At equal specificity the push rule wins
by source order, so fullscreen + push animates between mismatched
axes. The fade fix is clean.

The cart-bubble override is now gated by
`data-hp-cart-badge-override="true"`, default `false`, restoring
Dawn's color-scheme accent as the unconfigured-merchant path. The
new checkbox is positioned below the badge color settings in the
schema, which inverts the natural editing order (configure colors,
discover after the fact that they're gated).

The transparent-header offset gets a parallel
`:has(.header-wrapper--transparent)` selector alongside the JS-driven
`[data-hp-has-transparent="true"]` selector. `:has()` is Baseline
2023; older browsers fall through to the JS path. The validator's
coverage check now counts `hp_foo` + `--hp-foo` + `data-hp-foo` per
id; the `>= 1` total threshold is unchanged, so an id present only
in the Liquid body still passes.

The localization (#10) compromise migrates the bottom-nav account
visible label to `customer.account_fallback` / `customer.log_in`.
The wishlist labels, the bottom-nav `<nav aria-label>`, and all new
schema labels stay English by user instruction.

### Cart-count fix: receiver span missing on empty-cart boot

The MutationObserver on `#cart-icon-bubble` watches the persistent
`<a>` element (not the inner `.cart-count-bubble`, which Dawn removes
when the cart goes empty). Lifecycle: a module-scoped
`cartBubbleObserver`, defensively re-disconnected inside
`setupBottomNav` and torn down by `teardownPremiumHeader()` from both
`shopify:section:load` (before re-init) and `shopify:section:unload`.

The defect lives in the Liquid template:

```liquid
<span class="svg-wrapper hp-bottom-nav__cart-icon">
  <svg ...></svg>
  {%- if cart != empty -%}
    <span class="hp-bottom-nav__cart-count" aria-hidden="true">{{ cart.item_count }}</span>
  {%- endif -%}
</span>
```

If the storefront loads with an empty cart, the `.hp-bottom-nav__cart-count`
span is never emitted. The observer correctly fires on the first
add-to-cart (Dawn injects `.cart-count-bubble` into `#cart-icon-bubble`'s
inner HTML), but the sync handler bails:

```js
function syncBottomNavCartCount() {
  var bottomCount = document.querySelector('.hp-bottom-nav__cart-count');
  var cartIconBubble = document.getElementById('cart-icon-bubble');
  if (!bottomCount || !cartIconBubble) return;  // <-- bails forever
  ...
}
```

`bottomCount` is null, no recovery path, the bottom-nav cart count
stays hidden until the next full page load. The cleaner fix is to
always emit the span and let the existing `style.display` branch
handle the empty case (the JS already sets `display: none` for that
exact path).

### Push drawer animation: open-state collides with fullscreen

The v2 fix adds:

```css
.header-wrapper--premium[data-hp-mobile-anim="push"] menu-drawer[open] .menu-drawer,
.header-wrapper--premium[data-hp-mobile-anim="push"] details[open] > .menu-drawer {
  transform: translateX(0);
}
```

This pairs cleanly with the right-slide drawer (closed
`translateX(100%)`, open `translateX(0)`) and with default/left
positions where Dawn's stock left-slide is on the X axis. Fullscreen
is the odd one out:

```css
.header-wrapper--premium[data-hp-mobile-drawer="fullscreen"] .menu-drawer {
  transform: translateY(-100%);  /* closed */
}
.header-wrapper--premium[data-hp-mobile-drawer="fullscreen"] menu-drawer[open] .menu-drawer {
  transform: translateY(0);       /* open */
}
```

Specificity for fullscreen-open and push-open is equal (one attribute
selector + one element + one attribute selector + one class). At
equal specificity, source order wins; the push block is below the
position block, so push's `translateX(0)` overrides fullscreen's
`translateY(0)`. The drawer animates between `translateY(-100%)` and
`translateX(0)` - browsers interpolate each axis independently, so
the open drawer ends up at `translateX(0)` (which is `translateY(0)`
implicitly) and the close path reverses to `translateY(-100%)`,
working only because both targets resolve to the same on-screen
position. The slide direction is inconsistent though, and any future
drawer position introduced (slide-from-top, slide-from-bottom)
silently inherits push's X-axis assumption.

The cleaner shape is to either delegate push to whatever the
drawer-position rule sets (write per-position open-state rules, drop
push's `translateX(0)`), or scope the push open rule with
`:not([data-hp-mobile-drawer="fullscreen"])`.

### Cart-bubble override: gating works, schema ordering is awkward

The CSS rule and the data-attribute wire up correctly:

```css
.header-wrapper--premium[data-hp-cart-badge-override="true"] .cart-count-bubble {
  background-color: var(--hp-cart-badge-bg, #000);
  color: var(--hp-cart-badge-text, #fff);
  ...
}
```

Liquid emits `true` / `false` strings for checkbox values, the
`="true"` selector matches only the on state, and the off path
leaves Dawn's color-scheme accent untouched.

The schema ordering is the soft spot. The new checkbox sits AFTER
`hp_cart_badge_bg` and `hp_cart_badge_text`. A merchant adjusts the
badge colors, sees no change in preview, then has to scroll down to
find the override toggle. The `info` text on the checkbox is good
("Off by default, so flipping Premium Header on does not repaint
Dawn's color-scheme accent on the cart badge..."), but a merchant
who already adjusted the colors won't see that until they scroll.
Move the checkbox above the colors, or annotate each color setting
with an inline `info` line referencing the override.

### Transparent-header `:has()` fallback

```css
.shopify-section.section-header[data-hp-has-transparent="true"] + *,
.shopify-section.section-header:has(.header-wrapper--transparent) + * {
  margin-top: 0 !important;
}
```

`:has()` reached Baseline 2023 (Chrome/Edge 105 in 2022, Safari 15.4
in 2022, Firefox 121 in January 2024). For browsers older than
Firefox 121 / Safari 15.4, the `:has()` arm of the selector list
would historically have invalidated the entire rule under strict
CSS parsing; modern engines treat `:has()` as a forgiving pseudo and
ignore unrecognized selectors in a list, so the
`[data-hp-has-transparent="true"]` arm survives and continues to work
via the JS-driven path. Worth keeping in mind only if the storefront
needs to support pre-2024 Firefox or pre-2022 Safari, neither of
which is realistic in 2026.

### Lifecycle: `data-hp-initialized` removed; one detached-node edge case

`data-hp-initialized` was set-but-never-read (v1 #8) and is gone.
Re-init idempotency now relies on the module-scoped
`cartBubbleObserver` reference and `teardownPremiumHeader()`, which
disconnect on `shopify:section:load` (before re-init) and
`shopify:section:unload`. The hover-trigger handlers' `WeakMap`
keyed on parent `<li>` GCs naturally when old DOM is replaced.

Edge case: if a Dawn fork (or a future stock change) ever replaces
`#cart-icon-bubble` itself rather than its innerHTML, the observer
would still be watching the detached old node and never see updates
on the new one. Stock Dawn's cart.js / cart-notification.js replace
innerHTML, so this is theoretical for stock Dawn, but worth flagging
for forks.

### Validator coverage check

The coverage threshold is unchanged (`body + css + js >= 1`); what
changed is that each of those counts now includes `hp_foo`,
`--hp-foo`, and `data-hp-foo`. An id present only in the Liquid body
still passes, so no false-positive regression. The reality-check
report's per-id table is regenerated with non-zero css counts where
wiring genuinely exists, so the v1 misleading "css=0, js=0 for every
row" is gone.

### Sticky-footer side effect: documented, not scoped

`body:has(.hp-bottom-nav)` still applies a global 64px bottom padding
on mobile. The v2 commit chose documentation over scoping: a CSS
comment at the rule and an expanded `info` string on
`hp_show_mobile_bottom_nav` warn merchants that any sticky footer,
cookie banner, or chat widget pinned to `bottom: 0` on mobile gets
pushed up. The behavioral side effect is unchanged from v1.

### Localization (#10) status

Migrated to `t:` keys: the bottom-nav account `<a aria-label>` and
visible `<span>` label, both using `customer.account_fallback` /
`customer.log_in`.

Still English: the `<nav aria-label="Mobile bottom navigation">`
on the bottom-nav root (could reuse a Dawn key but currently doesn't),
the wishlist visible and `aria-label` strings (no Dawn-locale
equivalent), and all new `hp_*` schema labels and `info` strings
(deferred per the no-new-`t:`-keys rule).

### Not tested

- No live Shopify test of the empty-cart -> add-to-cart bottom-nav
  transition (would surface issue #1).
- No live test of the `fullscreen` + `push` drawer combination (would
  surface issue #2).
- No theme-editor test of the `shopify:section:load` /

- No unit test asserting the validator's three-form coverage check
