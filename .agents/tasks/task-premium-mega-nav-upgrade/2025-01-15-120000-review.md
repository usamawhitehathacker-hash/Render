# Premium mega-nav upgrade: bug fixes, dropdown behaviors, and 116 new settings

This change fixes two bugs (empty dropdown overlays causing page blackout, sticky header line jump) and adds 116 new Shopify theme customization settings across the mega-navigation and premium header sections. The settings span dropdown open behavior (TheJacketMaker-style animations, overlays, triggers), advanced hover effects (6 animation styles, scale, background highlight), navigation typography, badge enhancements, spacing controls, and mobile polish. The approach wires each setting through the Liquid schema, outputs corresponding CSS variables or data attributes, and implements CSS-driven behavior with minimal JS additions.

**Watch for:** Transform conflicts between contained/boxed dropdown centering and slide-down animations will visually break centered dropdowns (confirmed). The page overlay is trapped inside the mega-nav stacking context and cannot dim page content below the nav (confirmed). ~30 new settings are schema-only with no CSS/JS/HTML implementation (confirmed). The close animation setting is exposed but entirely non-functional (confirmed).

## High-level view

The bug fixes use layered defense: Liquid conditionals prevent empty dropdown divs from rendering, JS guards skip activation when no content exists, and a CSS `:empty` rule provides fallback. The sticky underline fix uses `will-change: transform` to stabilize pseudo-element positioning, and the top-line collapses with `height: 0` instead of `visibility: hidden`.

The dropdown open behavior system adds three trigger modes, five container animations, six content stagger variants, and a page overlay. The architecture is well-factored — CSS handles presentation via data-attribute selectors, JS only toggles classes and manages timing. Two geometry conflicts undermine it: slide-down/fade-slide animations use `transform` on the same element that contained/boxed centering uses `translateX(-50%)`, and the overlay is trapped in a stacking context.

Roughly 30 of the 116 declared settings are schema entries with no corresponding implementation. Dropdown layout variants (tabbed, grid-images, promo-left), close animations, mobile drawer styles, and various toggles exist as JSON settings only. They appear in the theme editor but produce no visual change.

The header layout additions (height, alignment, shadows, scroll behavior) are clean CSS driven by data attributes with no identified functional issues.

## Transform conflict: dropdown centering vs. container animations

The contained and boxed width modes position the dropdown with:
```css
left: 50%;
transform: translateX(-50%);
```

The slide-down-smooth animation applies to the same element:
```css
/* Inactive state */
transform: translateY(-10px);
/* Active state */
transform: translateY(0);
```

CSS `transform` is a single property — you cannot compose two separate declarations. When both selectors match, source order wins (animation declarations come later in the file), so the centering is lost. The dropdown snaps to `left: 50%` without the `-50%` offset correction, appearing shifted to the right by half its own width. The `height-grow` and `curtain-reveal` animations avoid this by using `clip-path` instead of `transform`, so they work correctly with contained/boxed modes.

## Overlay stacking context trap

The overlay is placed as a child of `.mega-navigation-section`:
```html
<div class="mega-navigation-section" style="z-index: 100; position: relative;">
  <div class="mega-nav-overlay" style="position: fixed; z-index: 99;"></div>
</div>
```

When a parent creates a stacking context (via `z-index` + positioning), all children participate in that context — even `position: fixed` elements. The overlay's z-index of 99 works within the section (behind the dropdown at 999), but the entire section is at z-index 100 relative to the page. Shopify themes commonly set z-index on announcement bars, other sections, and footers. Any page element with z-index > 100 won't be dimmed. The TheJacketMaker-style overlay needs to be outside any stacking context (direct child of `body` or injected via JS) to reliably cover page content.

## Non-functional and schema-only settings

`mn_close_animation` offers four options but nothing references `data-mn-close-animation` in CSS or JS. Closing uses CSS transition reversal implicitly, making "same-as-open-reverse" the only behavior — by accident.

`mn_dropdown_layout` declares six variants but only `columns-only` and `columns-promo-right` match the existing HTML structure. `tabbed`, `grid-images`, and promo-positional variants would require entirely different Liquid markup.

15 mobile settings (`mn_mobile_header_style`, `mn_mobile_menu_style`, `mn_mobile_nested_anim`, `mn_mobile_close_style`, `mn_mobile_show_logo`, `mn_mobile_footer_social`, `mn_mobile_bottom_icon_style`, `mn_mobile_bottom_labels`, `mn_mobile_bottom_active`) have no data attributes, CSS rules, or JS handlers anywhere.

## Scroll listener performance

```javascript
window.addEventListener('scroll', function () {
  var activeItems = section.querySelectorAll('.mega-nav-item.is-active');
  if (activeItems.length > 0) {
    closeAllDropdowns(section);
  }
}, { passive: true });
```

This queries the DOM on every scroll frame (60+ times/second) even when no dropdown is open. Once the first scroll closes the dropdown, subsequent events are pure waste. Short-circuit with a flag or remove the listener after first trigger.

## Swipe-to-close attribute mismatch

```javascript
var drawerPosition = drawer.getAttribute('data-position') || 'left';
```

The Liquid template sets `data-mn-mobile-drawer-position` on the `.mega-navigation-section` div, not `data-position` on the drawer element. Unless the drawer has a separate `data-position` attribute set elsewhere in the pre-existing code, this always resolves to `'left'`. Right-side drawers would need a leftward swipe to close (which the `drawerPosition === 'right' && diffX > 0` branch handles), but since `drawerPosition` is always `'left'`, only leftward swipes (diffX < 0) trigger close — the opposite of what a right-side drawer user would expect.

## Style-attribute selectors vs. data attributes

The CSS uses two different patterns for setting-driven behavior:

- **Data attributes** (used by animations, width, hover effects): `[data-mn-container-animation="slide-down-smooth"]` — robust, whitespace-insensitive, inspectable in DevTools.
- **Style attribute substring match** (used by shadows, gradients, easing, alignment, separators): `[style*="--mn-dropdown-shadow-style: subtle"]` — breaks if Liquid outputs extra whitespace, if the property is removed, or if minification changes formatting.

The inconsistency means some features are more fragile than others. Shadow styles, gradient directions, hover easing, and column separators all use the fragile pattern.
